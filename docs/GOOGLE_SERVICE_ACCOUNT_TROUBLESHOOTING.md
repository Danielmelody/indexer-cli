# Google Service Account Troubleshooting | Google 服务账户故障排查

## "Not enough private keys in PEM" Error

### Root Cause (原因)

This error occurs when the service account JSON file contains a private key in **PKCS#1 format** (`BEGIN RSA PRIVATE KEY`) instead of the required **PKCS#8 format** (`BEGIN PRIVATE KEY`).

此错误表示服务账户 JSON 文件中的私钥为 **PKCS#1 格式**（`BEGIN RSA PRIVATE KEY`），而非所需的 **PKCS#8 格式**（`BEGIN PRIVATE KEY`）。

### Expected Format (正确格式)

**Correct (PKCS#8)**:
```
-----BEGIN PRIVATE KEY-----
MIIEvQIBADANBgkqhkiG9w0BAQEF...
-----END PRIVATE KEY-----
```

**Incorrect (PKCS#1)**:
```
-----BEGIN RSA PRIVATE KEY-----
MIIEpAIBAAKCAQEA2Z1JqSy9f...
-----END RSA PRIVATE KEY-----
```

### Why This Happens (发生原因)

1. **Old Creation Method**: Key was created using an older API or tool（旧版 API 或工具创建）
2. **Manual Editing**: Key was manually converted or edited（手动转换或编辑）
3. **Third-Party Tools**: Key was generated by non-Google tools（非 Google 工具生成）
4. **Test/Mock Data**: Using test data instead of real Google-generated keys（使用测试/模拟数据）

### Solution: Create a New Key (解决方案：创建新密钥)

**Google Cloud Console Method** (Recommended)（推荐）:

1. **Visit Service Accounts**:
   ```
   https://console.cloud.google.com/iam-admin/serviceaccounts
   ```

2. **Select Your Project**:
   - Make sure you're in the correct project（确保在正确的项目）
   - Look for your service account in the list（在列表中找到你的服务账户）

3. **Create New Key**:
   - Click on the service account name（点击服务账户名称）
   - Go to the "KEYS" tab（转到“密钥”标签页）
   - Click "ADD KEY" → "Create new key"（点击“添加密钥”→“创建新密钥”）
   - **IMPORTANT**: Select "JSON" format (NOT P12)（重要：选择 JSON 格式，而非 P12）
   - Click "CREATE"（点击“创建”）

4. **Verify the Downloaded File** (验证下载的文件):
   ```bash
   cat ~/Downloads/your-key.json | jq -r '.private_key' | head -1
   ```

   Should show: `-----BEGIN PRIVATE KEY-----` ✓
   NOT: `-----BEGIN RSA PRIVATE KEY-----` ✗

### What Google Actually Generates (Google 实际生成的内容)

When you create a JSON service account key through the official Google Cloud Console:

通过官方 Google Cloud Console 创建 JSON 服务账户密钥时：

- ✅ **Always** uses PKCS#8 format (`BEGIN PRIVATE KEY`)（始终使用 PKCS#8 格式）
- ✅ **Never** uses PKCS#1 format (`BEGIN RSA PRIVATE KEY`)（绝不使用 PKCS#1 格式）
- ✅ Compatible with all Google APIs and client libraries（兼容所有 Google API 和客户端库）

If you see `BEGIN RSA PRIVATE KEY`, the key was **NOT** created through the standard Google Cloud Console JSON download process.

如果看到 `BEGIN RSA PRIVATE KEY`，该密钥**并非**通过标准 Google Cloud Console JSON 下载流程创建。

### Manual Conversion (Last Resort) (手动转换：最后手段)

If you absolutely cannot get a new key from Google, you can convert PKCS#1 to PKCS#8:

如果实在无法从 Google 获取新密钥，可以将 PKCS#1 转换为 PKCS#8：

```bash
# Extract the private key from JSON
cat your-file.json | jq -r '.private_key' > old_key.pem

# Convert PKCS#1 to PKCS#8
openssl pkcs8 -topk8 -inform PEM -outform PEM \
  -in old_key.pem -out new_key.pem -nocrypt

# Update the JSON file
# Manually replace the "private_key" field value with the content of new_key.pem
# Make sure to escape newlines as \n in the JSON
```

**Warning**: Manual conversion is error-prone. **Always prefer creating a new key** from Google Cloud Console.

**警告**：手动转换容易出错。**始终优先从 Google Cloud Console 创建新密钥**。

### Add to Google Search Console (添加到 Google Search Console)

After getting a valid key, remember to add the service account to Google Search Console:

获得有效密钥后，记得将服务账户添加到 Google Search Console：

1. Visit: https://search.google.com/search-console
2. Select your website property（选择你的网站属性）
3. Settings → Users and permissions → Add user（设置→用户和权限→添加用户）
4. Email: `your-sa@project-id.iam.gserviceaccount.com` (from JSON)（邮箱来自 JSON）
5. Permission: **Owner** (required for Indexing API)（权限：所有者，Indexing API 必需）

## Common Questions (常见问题)

**Q: Why does Google use PKCS#8?**（为什么 Google 使用 PKCS#8？）

A: PKCS#8 is the modern standard that includes algorithm information within the key, making it more versatile and compatible with various APIs.

答：PKCS#8 是现代标准，在密钥中包含算法信息，使其更通用、兼容更多 API。

**Q: Can I use P12 format instead?**（可以用 P12 格式代替吗？）

A: P12 is deprecated by Google. Always use JSON format for new keys.

答：P12 已被 Google 弃用。新密钥请始终使用 JSON 格式。

**Q: How do I know if my key is from Google?**（如何知道密钥是否来自 Google？）

A: Real Google-generated JSON keys always have:
- `BEGIN PRIVATE KEY` (PKCS#8 format)
- Valid `client_email` ending in `.iam.gserviceaccount.com`
- `auth_uri` pointing to `accounts.google.com`
- `token_uri` pointing to `oauth2.googleapis.com`

答：真正的 Google 生成 JSON 密钥始终具有：
- `BEGIN PRIVATE KEY`（PKCS#8 格式）
- 有效的 `client_email`，以 `.iam.gserviceaccount.com` 结尾
- `auth_uri` 指向 `accounts.google.com`
- `token_uri` 指向 `oauth2.googleapis.com`

**Q: My key still doesn't work after creating a new one**（创建新密钥后仍然无法使用）

A: Check these:
1. Wait 3-5 minutes after enabling the Indexing API（启用 Indexing API 后等待 3-5 分钟）
2. Ensure service account is added to Search Console as **Owner**（确保服务账户以“所有者”身份添加到 Search Console）
3. Verify you're using the latest JSON file you just downloaded（确认使用的是刚下载的最新 JSON 文件）
4. Make sure the file hasn't been edited or corrupted（确保文件未被编辑或损坏）

## Configuration File Issues (配置文件问题)

**Q: After running setup, verify still shows the old/wrong file**（运行 setup 后，verify 仍显示旧/错误文件）

A: This can happen if you have an old `indexer.yaml` file. Solution:

答：如果存在旧的 `indexer.yaml` 文件，可能会发生这种情况。解决方案：

1. Delete or backup the old config（删除或备份旧配置）:
   ```bash
   rm indexer.yaml  # or: mv indexer.yaml indexer.yaml.backup
   ```

2. Re-run setup with your file（使用你的文件重新运行 setup）:
   ```bash
   indexer-cli google setup --service-account ~/path/to/your-file.json
   ```

3. Verify the config was saved correctly（验证配置是否正确保存）:
   ```bash
   cat indexer.yaml | grep service_account_file
   ```

   Should show your file path under `auth.service_account_file`

**Q: Why does the config have two places for service_account_file?**（为什么配置中 service_account_file 有两个位置？）

A: For backward compatibility:
- `google.auth.service_account_file` - New location (recommended)（新位置，推荐）
- `google.service_account_file` - Legacy location（旧位置）

The verify command checks the new location first, then falls back to legacy.

答：为了向后兼容。verify 命令优先检查新位置，然后回退到旧位置。
